Modelling people's names (is hard)

Everyone has a name... or multiple names. What we are called (usually) depends on the environment we are in. I myself use a nickname at work (Kudłaty, or Shaggy in English -- I hope you watched Scooby Doo), another nickname on the internet and (most of) my personal life. I also use my legal name in some contexts (e.g. for tax and another official purposes). Most of the people that I know from the internet use a nickname (have you ever seen anyone using their legal name on IRC, or a discussion forum?). All of this stuff, that is easy for humans to grasp is (as always) very compliacted to model in computers.

Let's start with a very simple model:

	class PersonalName
	{
		public string Firstname { get; set; }
		public string Surname { get; set; }
	}

Seems reasonable, right? Well, it is not... What about the Japanese emperor -- Akihito (yup, that's his whole name)? Our model covers some cases of western names, but isn't really exhaustive.

So, next model:

	interface IPersonalName
	{
		public string DisplayName { get; }
	}
	
	class PolynymousName : IPersonalName
	{
		public string Firstname { get; set; }
		public string Surname { get; set; }
		
		public string DisplayName
		{
			get { return string.Format("{0} {1}", Firstname, Surname); }
		}
	}
	
	class MononymousName : IPersonalName
	{
		public string DisplayName { get; set; }
	}

But we have uncounsiously introduced an assumption that first name always precedes the last name. This is not true, for example in China. Let's try again

	interface IPersonalName
	{
	}
	
	class PolynymousName : IPersonalName
	{
		public string Firstname { get; set; }
		public string Surname { get; set; }
	}
	
	class MononymousName : IPersonalName
	{
		public string Name { get; set; }
	}

	interface IPersonalNameFormatter<T> where T : IPersonalName
	{
		public string Format(T name);
	}
	
	class WesternNameFormatter : IPersonalNameFormatter<PolynymousName>
	{
		public string Format(PolynymousName name)
		{
			return string.Format("{0} {1}", name.Firstname, name.Surname);
		}
	}
	
	class ChineseNameFormatter : IPersonalNameFormatter<PolynymousName>
	{
		public string Format(PolynymousName name)
		{
			return string.Format("{0} {1}", name.Surname, name.Firstname);
		}
	}

That's quite a lot of code to represent a name. But is it sufficient? Of course not! There are obvious things -- probably China isn't the only country where the surname is written before the firstname. So maybe the formatters should have less specific names, like "FirstnameFirstFormatter", "LastnameFirstFormatter". We actually should probably use the term "Given name" , as the "Firstname" certainly isn't "first" here. There is also the issue of people who don't want to be called by the name that is defined in their documents. Some of them are transgender, some just are not used to being called that and there are probably more reasons, that I haven't even thought of. In some cases it is okay to just store preferred name, but sometimes (e.g. when processing card transactions) the official name is required. So let's define a Person class and see how it goes:

	class Person
	{
		IPersonalName OfficialName { get; set; }
		IPersonalName PrefferedName { get; set; }
	}

The interesting fact is that the preffered name and official name doesn't have to be of the same kind. I know a lot of people who have polynymous official names, but go by a mononymous preffered name (that includes myself -- my official name is Paweł Łuczkiewicz, but people just call me agares, or Kudłaty). So, is this model complete? I'm sure it's not. But I believe it's better than the usual "first name, last name", that is used all over the internet. So, next time you have to model a person's name, please think about all those people who don't fit the "95% of Western people" case.
